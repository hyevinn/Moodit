## 1. 프로젝트 개요

### 1.1 프로젝트 목표
> Moodit(무드잇)은 사용자의 소비 정보를 기반으로 소비 유형과 소비 성향을 분석하여 자신의 소비 습관을 직관적으로 이해할 수 있도록 돕는 AI 기반 소비 분석 서비스이다.
기존 가계부 앱의 단순 기록 기능을 넘어, 소비 이유와 소비 특징을 함께 분석하고 시각화하여 사용자가 자신의 소비 패턴을 쉽고 흥미롭게 파악할 수 있도록 하는 것을 목표로 한다.


### 1.2 대상 사용자
> 자신의 소비 습관과 소비 성향을 분석하고 싶은 대학생
> 감정 및 소비 이유 기반 소비 패턴을 확인하고 싶은 사용자
> AI 소비 성향 분석 결과를 확인하고 싶은 사용자 


### 1.3 핵심 가치
> 소비 금액 기록보다 소비 행동 분석에 집중한다.
> 소비 이유와 소비 카테고리를 함께 분석한다.
> AI 기반 소비 성향 리포트를 제공한다.

## 2. 기능 요구사항 (Functional Requirements)

> 시스템이 *무엇을 하는가*를 정의한다. 최소 5개 이상 작성
> 문장 패턴: "사용자는 ~할 수 있다" / "시스템은 ~한다"

### FR-00: [위치 정보 수집 기능]

- **설명**: 사용자의 위치 정보를 수집한다.
- **수용 조건**:
  - [ ] 사용자의 위치 정보 수집 권한 허용 여부를 확인한다.
  - [ ] 위치 정보를 확인한다.

### FR-01: [알림 수신 동의 기능]

- **설명**: 사용자가 알림 수신에 동의한다.
- **수용 조건**:
  - [ ] 알림 수신 동의 버튼을 클릭할 수 있다.
  - [ ] 사용자의 알림 권한 허용 여부를 확인한다.
  - [ ] 알림을 수신한다.

### FR-02: [홈 화면 제공 기능]

- **설명**: 시스템은 서비스 소개 및 분석 시작 화면을 제공한다.
- **수용 조건**:
  - [ ] 홈 화면은 서비스 소개 및 분석 시작 화면을 제공한다.
  - [ ] 홈 화면에서 '소비 기록하기' 버튼을 클릭하면 소비 입력 화면으로 이동한다.
  - [ ] 홈 화면에 최근 분석 결과를 표시한다.
  - [ ] 홈 화면에 최근 선택한 소비 정보를 표시한다.
  - [ ] 앱 종료 후 다시 실행해도 최근 분석 결과가 유지되어 표시된다.
  
### FR-03: [선택형 소비 입력]

- **설명**: 사용자는 선택형 UI를 통해 소비 정보를 간단하게 입력할 수 있다.
- **수용 조건**:
  - [ ] 사용자는 소비 카테고리를 선택할 수 있다.
  - [ ] 사용자는 소비 금액 범위를 선택할 수 있다.
  - [ ] 사용자는 소비 이유를 선택할 수 있다.

### FR-04: [소비 정보 분석 요청 기능]

- **설명**: 사용자는 입력한 소비 정보를 기반으로 소비 성향 분석을 요청할 수 있다.
- **수용 조건**:
  - [ ] 분석 시작 버튼을 통해 소비 분석을 요청할 수 있다.
  - [ ] 입력한 소비 정보가 분석 화면으로 전달된다.
  - [ ] GroqCloud API가 호출된다.
  - [ ] 분석 중 로딩 상태가 표시된다.

### FR-05: [AI 소비 성향 분석 기능]

- **설명**: 시스템은 사용자의 소비 정보를 기반으로 소비 유형과 소비 성향을 분석한다.
- **수용 조건**:
  - [ ] 소비 카테고리와 소비 이유를 분석에 활용한다.
  - [ ] GroqCloud API를 활용하여 소비 성향 리포트를 생성한다.
  - [ ] 소비 특징과 소비 습관을 자연어 형태로 생성한다.

### FR-06: [소비 리포트 제공 기능]

- **설명**: 사용자는 AI가 생성한 소비 성향 리포트를 확인할 수 있다.
- **수용 조건**:
  - [ ] 분석 결과가 텍스트 형태로 출력된다.
  - [ ] 소비 특징 및 소비 습관 설명이 포함된다.
  - [ ] 사용자는 소비 분석 결과를 화면에서 확인할 수 있다.

### FR-07: [결과 화면 시각화 기능]

- **설명**: 시스템은 소비 분석 결과를 시각적으로 제공한다.
- **수용 조건**:
  - [ ] 사용자가 입력한 소비 정보를 카드 형태로 표시한다.
  - [ ] 소비 유형을 카드 형태로 표시한다.
  - [ ] 소비 성향 키워드를 태그 형태로 표시한다.


### FR-08: [결과 화면 통합 제공 기능]

- **설명**: 시스템은 소비 분석 결과를 하나의 화면에서 제공한다.
- **수용 조건**:
  - [ ] 소비 정보 카드가 표시된다.
  - [ ] 소비 유형 카드가 표시된다.
  - [ ] 소비 성향 태그가 표시된다.
  - [ ] AI 분석 리포트가 표시된다.

### FR-09: [테마 전환 기능]

- **설명**: 사용자는 앱의 화면 테마를 다크 모드 또는 라이트 모드로 전환할 수 있다.
- **수용 조건**:
  - [ ] 다크 모드와 라이트 모드 전환 기능을 제공한다.
  - [ ] 테마 변경 시 전체 UI 색상이 변경된다.
  - [ ] 사용자는 현재 적용된 테마 상태를 확인할 수 있다.
  - [ ] 테마 설정이 앱 실행 간 유지된다.
  - [ ] 다크 모드와 라이트 모드 모두에서 UI가 정상적으로 표시된다.

## 3. 비기능 요구사항 (Non-Functional Requirements)

> *어떻게 동작하는가*를 정의한다. 직접 보이지 않지만 품질을 결정한다.

### 3.1 성능 (Performance)

- 앱 실행 후 5초 이내에 메인 화면 및 주요 UI를 표시해야 한다.
- 소비 분석 결과는 요청 후 5초 이내에 출력한다.

### 3.2 보안 (Security)

- 사용자의 개인정보를 저장하지 않는다.
- GroqCloud API Key는 BuildConfig 또는 Local Properties를 통해 관리한다.
- API Key가 소스 코드나 UI에 직접 노출되지 않아야 한다.

### 3.3 사용성 (Usability)

- 사용자는 선택형 UI를 통해 소비 카테고리, 소비 금액 범위, 소비 이유를 입력할 수 있어야 한다.
- 버튼 선택 시 색상 변화를 통해 현재 선택 상태를 확인할 수 있어야 한다.
- 분석 결과는 카드 UI와 태그 UI를 통해 직관적으로 제공되어야 한다.
- 사용자는 다크 모드와 라이트 모드를 자유롭게 전환하여 사용할 수 있어야 한다.
- 다크 모드와 라이트 모드 모두에서 UI가 정상적으로 표시되어야 한다.
- 초보 사용자도 쉽게 사용할 수 있어야 한다.
- 사용자는 앱 재실행 후에도 최근 분석 결과를 확인할 수 있어야 한다.

### 3.4 호환성 (Compatibility)

- Android API 24 이상 환경에서 동작해야 한다.
- Jetpack Compose 기반으로 구현해야 한다.
- 다양한 Android 화면 크기에서 UI가 정상적으로 표시되어야 한다.

### 3.5 기타

- MVP 범위 내 핵심 기능 중심으로 구현한다.
- 유지보수가 쉽도록 기능별 모듈 구조를 적용한다.
- API 호출 로직을 별도 클래스로 관리한다.
